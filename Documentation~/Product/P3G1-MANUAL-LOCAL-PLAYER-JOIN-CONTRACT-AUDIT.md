# P3G.1 — Manual Local Player Join Contract Audit

## Objective

Close `DG-P3-03` before introducing Unity provisioning side effects.

This audit defines the minimum `LocalPlayerJoinRequest`, `LocalPlayerJoinResult`, Pending Join correlation and rollback semantics required to connect the Session-scoped Player participation runtime to `PlayerInputManager` in manual join mode.

## Type

Technical/runtime integration audit and architecture decision.

## Existing reusable runtime

### PlayerParticipationRuntimeContext

Already owns:

```text
ordered configured Slot roster
dynamic capacity
join window
atomic first-available reservation
Session-scoped reservation tokens
TryReleaseReservation
TryMarkJoined
foreign/stale token rejection
immutable snapshots
```

The provisioning bridge must consume these operations. It must not create a parallel Slot list or reservation registry.

### PlayerParticipationRuntimeHostModule

Already establishes the Session lifetime and typed access path from a known `FrameworkRuntimeHost` reference.

The provisioning bridge must be composed under this same Session owner. Caller-provided context ids are rejected as redundant and unsafe.

### Existing Unity input lane

The package already treats Unity `PlayerInput` and `PlayerInputManager` as technical integration evidence and keeps InputMode application separate from Player creation.

P3G must reuse that separation:

```text
Provisioning
  creates and admits the local Player host.

InputMode adapters
  later apply explicit input mode requests to an admitted PlayerInput.
```

## Decision

### Ordinary request

```text
LocalPlayerJoinRequest
  string Source
  string Reason
  InputDevice PairWithDevice?       // optional technical hint
  string ControlScheme              // optional, empty means unspecified
```

The request does not name a Slot. Allocation remains:

```text
First Available By Configured Order
```

The request does not contain Actor selection. `ActorProfile` belongs to P3H.

The request does not contain `playerIndex` or `splitScreenIndex`. The first product path has no requirement to control them, and `playerIndex` must remain diagnostic evidence rather than product identity.

### Request identity

A typed `LocalPlayerJoinOperationId` is generated by the provisioning bridge, not authored by the caller.

Recommended stable diagnostic shape:

```text
local-player-join:<session-context>:<sequence>
```

The operation id is not a global lookup key. It exists inside the Session-scoped bridge and result evidence.

### Pending operation

The bridge stores at most one synchronous provisioning operation in flight for the initial cut.

```text
PendingLocalPlayerJoin
  OperationId
  Request
  PlayerSlotReservationToken
  expected PlayerInputManager reference
  state
  direct PlayerInput result
  joined callback PlayerInput evidence
```

Serializing join execution avoids ambiguous callback correlation and matches the current single Session manager authority. Parallel joins may be reconsidered only after a real product requirement.

### Result

```text
LocalPlayerJoinResult
  LocalPlayerJoinStatus Status
  LocalPlayerJoinOperationId OperationId
  LocalPlayerJoinRequest Request
  PlayerParticipationOperationResult ReservationResult
  PlayerParticipationOperationResult CommitResult
  PlayerParticipationOperationResult RollbackResult
  PlayerSlotRuntimeSnapshot Slot
  PlayerInput PlayerInput
  PlayerActorDeclaration PlayerActorDeclaration
  int UnityPlayerIndex
  LocalPlayerJoinCallbackConfirmation CallbackConfirmation
  string Message
```

The result exposes Unity objects as technical evidence because this is the Unity provisioning boundary. Pure product code should consume Slot/session snapshots rather than use `playerIndex` as identity.

## Status model

Required semantic outcomes:

```text
SucceededJoined
RejectedInvalidRequest
RejectedOperationInFlight
RejectedRuntimeUnavailable
RejectedManagerUnavailable
RejectedManagerConfiguration
RejectedJoiningClosed
RejectedCapacityReached
RejectedNoAvailableSlot
RejectedProvisioningReturnedNull
RejectedUnexpectedJoin
RejectedCorrelationMismatch
RejectedMissingPlayerInput
RejectedMissingPlayerActorDeclaration
RejectedForeignOrStaleReservation
FailedAdmission
FailedRollback
```

The implementation may group P3F reservation rejections through a mapped status helper, but the final result must preserve the original `PlayerParticipationOperationResult`.

## Synchronous correlation policy

The implementation must support either callback order:

```text
A. joined callback -> JoinPlayer return
B. JoinPlayer return -> joined callback
```

Required rule:

```text
direct returned PlayerInput and callback PlayerInput must be the same reference
```

The bridge may complete admission from the direct non-null return without waiting one frame. Callback absence at the exact return boundary is recorded as pending confirmation, not solved with an arbitrary delay.

A callback that arrives for another PlayerInput while a pending operation exists is divergence and blocks admission.

## Failure and rollback matrix

| Failure | Slot action | Unity host action | Result evidence |
|---|---|---|---|
| request invalid | none | none | rejected request |
| joining/capacity/no Slot | none beyond P3F result | none | mapped P3F rejection |
| JoinPlayer returns null | release reservation | none | null provisioning + rollback |
| callback mismatch | release reservation | reject created host | correlation mismatch + rollback |
| missing PlayerInput | release reservation | reject created host | missing evidence + rollback |
| missing PlayerActorDeclaration | release reservation | reject created host | missing declaration + rollback |
| TryMarkJoined fails | release reservation when token remains valid | reject created host | admission failure + rollback |
| rollback fails | preserve original failure | reject/quarantine host | `FailedRollback` with both diagnostics |

No failure may leave an invisible `Reserved` Slot.

## PlayerInputManager validation

Before reserving a Slot, validate:

```text
explicit manager reference is present
manager uses manual join behavior
player prefab is configured
manager is the Session-authorized manager
technical maxPlayerCount allows another Player when enabled
```

The manager's serialized technical ceiling is read-only runtime policy. P3G reads it; it does not mutate it or use reflection.

## Authoring surface required by implementation

The future authoring component should be designer-first:

```text
Local Player Provisioning
  PlayerInputManager reference
  Current technical ceiling summary
  Manual Join validation
  Player prefab evidence
  Session Slot count comparison

Advanced / Debug
  active Pending Join
  reservation token
  operation id
  callback/direct evidence
  last result and rollback
```

The authoring component must not call `JoinPlayer` from `OnEnable`, `Start`, validation or Apply/Rebuild.

## Runtime ownership

```text
FrameworkRuntimeHost / Session
  owns PlayerParticipationRuntimeContext
  owns LocalPlayerProvisioningBridge
  owns one explicit PlayerInputManager reference

PlayerInputManager
  provisions the Unity local Player root

LocalPlayerProvisioningBridge
  orchestrates reservation, correlation, validation, commit and rollback

Activity
  later evaluates participation requirements and gameplay readiness
```

## Files that must not absorb this responsibility

```text
GameFlowRuntime
RouteLifecycleRuntime
ActivityFlowRuntime
PlayerControlRuntimeContext
RuntimeContentRuntime
PlayerComposer
InputMode PlayerInput adapters
```

They may consume admitted evidence later; none becomes the join authority.
