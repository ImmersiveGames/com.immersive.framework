# PlayerComposer QA Readiness Plan

Status: QA readiness plan, no QA implementation
Date: 2026-07-09
Package: `com.immersive.framework`
Cut: P1G

## 1. Objective

Define the minimum QA scope for the official `PlayerComposer` Product Surface after the package MVP passed in FIRSTGAME.

This document does not implement QA. It defines what `QAFramework` should prove once the QA cut starts.

The goal is to move from manual proof to repeatable technical validation without making QA the product UX.

## 2. Context

`PlayerComposer` is now the first validated Product Surface of the Immersive Framework 1.0.

Validated flow:

```text
PlayerComposer official package component
+ Inspector Validate
+ Inspector Apply/Rebuild
+ technical materialization
+ diagnostics breakdown
+ FIRSTGAME proof
```

FIRSTGAME confirmed:

```text
configured official PlayerComposer intent
Validation succeeded
Apply/Rebuild completed with blocked='0'
second Apply/Rebuild completed with created='0' repaired='0'
resetEnabled='False'
resetParticipantPolicy='None'
legacy local player binding facade removed
```

## 3. QA role

QAFramework must validate the official contract created by the package.

QAFramework must not define the UX and must not replace the `PlayerComposer` authoring flow.

Correct order:

```text
Package defines Product Surface.
FIRSTGAME proves real usability.
QAFramework proves repeatable technical behavior.
```

## 4. Scope for first QA cut

The first QA cut should validate only the MVP surface that already exists.

In scope:

- `PlayerComposer` can materialize the minimum technical bindings.
- `PlayerComposer` Apply/Rebuild is idempotent.
- missing required `PlayerInput` blocks Apply/Rebuild.
- default reset settings do not create `UnityTransformResetParticipant`.
- optional transform reset policy creates transform participant only when requested.
- drift repair updates generated fields without duplicating components.
- generated identity uses typed actor/slot values, not object names or paths.
- generated objects remain under the expected Player-local topology.

Out of scope:

- `PlayerRecipe`.
- `PlayerRuntimeContext`.
- movement.
- spawn.
- multiplayer join.
- save/progression.
- camera composer.
- FIRSTGAME-specific IDs, paths or scripts.
- recreating the removed FIRSTGAME player binding facade.

## 5. Required QA cases

### QA-PC-001 - Apply creates MVP bindings

Purpose: prove that a fresh authored Player can be materialized by the official `PlayerComposer`.

Setup:

```text
GameObject Player
  PlayerComposer
  PlayerInput
```

Composer settings:

```text
actorId = qa.player.actor
playerSlotId = player.1
inputBindingRequired = true
cameraBindingRequired = true
resetEnabled = false
resetParticipantPolicy = None
```

Expected result:

```text
Apply/Rebuild succeeds
blocked = 0
created > 0
_Framework/_Bindings exists
required root components exist
required generated binding components exist
reset transform participant is not created
```

### QA-PC-002 - Rebuild is idempotent

Purpose: prove repeated Apply/Rebuild does not duplicate or repair unnecessarily.

Setup:

Use the result of QA-PC-001.

Action:

```text
Run Apply/Rebuild twice without changing the object.
```

Expected result on second run:

```text
created = 0
repaired = 0
blocked = 0
alreadyValid > 0
no duplicate _Framework
no duplicate _Bindings
no duplicate generated binding components
```

### QA-PC-003 - Missing PlayerInput blocks when input is required

Purpose: prove required configuration fails explicitly.

Setup:

```text
GameObject Player
  PlayerComposer
```

Composer settings:

```text
inputBindingRequired = true
```

Expected result:

```text
Validate fails
Apply/Rebuild fails
blocking issue mentions PlayerInput
no silent fallback
no generated binding pretending success
```

### QA-PC-004 - Reset default is optional

Purpose: prove `PlayerComposer` does not make reset participants mandatory.

Setup:

```text
GameObject Player
  PlayerComposer
  PlayerInput
```

Composer settings:

```text
resetEnabled = false
resetParticipantPolicy = None
```

Expected result:

```text
Apply/Rebuild succeeds
UnityTransformResetParticipant is not generated
UnityResetSubjectAdapter is not generated by default reset policy
skippedByPolicy includes reset-related optional materialization
```

### QA-PC-005 - Transform reset policy is optional and idempotent

Purpose: prove transform reset is available only when requested.

Setup:

```text
GameObject Player
  PlayerComposer
  PlayerInput
```

Composer settings:

```text
resetEnabled = true
resetParticipantPolicy = Transform
```

Expected result:

```text
Apply/Rebuild succeeds
UnityResetSubjectAdapter exists
UnityTransformResetParticipant exists
Transform participant target is explicit when generated under _Framework/_Bindings
second Apply/Rebuild does not duplicate reset subject or participant
```

### QA-PC-006 - Drift repair works

Purpose: prove generated materialization can be repaired from Composer intent.

Setup:

Use a materialized Player from QA-PC-001.

Drift example:

```text
change generated expectedPlayerSlotId away from composer playerSlotId
or clear a generated PlayerInput reference
or clear a generated camera target reference
```

Expected result:

```text
Apply/Rebuild succeeds
repaired > 0
created = 0 unless a component was intentionally removed
blocked = 0
second Apply/Rebuild returns repaired = 0
```

### QA-PC-007 - Removed generated component is recreated

Purpose: distinguish drift repair from missing generated component recovery.

Setup:

Use a materialized Player from QA-PC-001.

Action:

```text
Delete one generated binding component under _Framework/_Bindings.
Run Apply/Rebuild.
```

Expected result:

```text
created > 0
blocked = 0
missing generated component is restored
no duplicate unrelated components
```

### QA-PC-008 - Identity does not come from GameObject name/path

Purpose: prove actor/slot identity remains explicit and typed.

Setup:

Use a materialized Player from QA-PC-001.

Action:

```text
Rename Player GameObject.
Move Player under another parent.
Run Validate and Apply/Rebuild.
```

Expected result:

```text
actorId remains qa.player.actor
playerSlotId remains player.1
Validate succeeds
Apply/Rebuild succeeds
no generated field changes to GameObject name or path as identity
```

## 6. Optional later QA cases

These cases are useful but should not block the first QA cut:

- invalid action map if QA can reliably create an invalid state;
- multiple conflicting generated bindings;
- player slot occupancy conflict with another actor source;
- camera target required with creation policy disabled;
- serialization reload after domain reload;
- prefab variant behavior;
- multi-player local authoring;
- interaction with future `PlayerRecipe`.

## 7. QA implementation direction

The first QA implementation should prefer synthetic QA assets, not FIRSTGAME assets.

Recommended location:

```text
Assets/ImmersiveFrameworkQA/PlayerComposer/
```

Recommended shape:

```text
PlayerComposerQaScenario
PlayerComposerQaCanvas or menu entry
minimal test scene or generated temporary scene
positive and negative scenario methods
clear logs with QA-PC case ids
```

The QA should execute the same authoring actions the product surface supports. It should not create internal bindings manually as its primary path.

## 8. Expected QA log shape

QA logs should expose diagnostics in a compact form.

Example:

```text
[INFO][Immersive.Framework][PlayerComposerQa] QA-PC-002 passed. player='QA Player' created='0' repaired='0' alreadyValid='13' skippedByPolicy='2' blocked='0'.
```

For failures:

```text
[WARN][Immersive.Framework][PlayerComposerQa] QA-PC-003 passed negative case. issue='PlayerComposer requires PlayerInput when input binding is required.'.
```

## 9. Acceptance criteria for first QA cut

The first QA implementation is PASS if:

- at least QA-PC-001, QA-PC-002, QA-PC-003 and QA-PC-004 are automated;
- QA-PC-005 is automated if reset policy fields are stable enough;
- no FIRSTGAME script or asset is required;
- no removed FIRSTGAME facade is recreated;
- QA uses the official `PlayerComposer` surface;
- positive cases end with blocked = 0;
- idempotent cases end with created = 0 and repaired = 0 on second Apply/Rebuild;
- negative cases fail explicitly with diagnostic issue text;
- QA does not treat smoke as user-facing authoring UX.

## 10. Product boundary

QA validates the surface. It does not own it.

The official user path remains:

```text
Add PlayerComposer
Configure intent
Validate
Apply/Rebuild
Inspect Advanced/Debug if needed
```

QA exists to prevent regressions after this path has been defined and proven.
