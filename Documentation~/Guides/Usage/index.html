Immersive Framework
Preview 12

Guia prático para o fluxo mínimo jogável validado até o Preview 12: Menu, Gameplay, Activity A/B, Loading, Transition Gate, Pause visual, Pause por botão, Pause por tecla via Global/Pause, pausa real via TimeScale, PlayerInput Gate Adapter, Object Reset, Reset Group e Activity Restart com uma única transição visual.

✓ FIRSTGAME Usage Model PASS v1.0.0-preview.12 Git Package Preview Unity 6.5
💡
Para quem é este guia

Este guia é para game designers trabalhando em um projeto consumidor limpo. Ele assume que o package com.immersive.framework já está instalado via Git e que você quer montar um fluxo jogável mínimo sem escrever código.

Escopo do Preview 12

O v1.0.0-preview.12 consolida o uso real do framework como package Git em projeto consumidor. Route/Activity podem bloquear requests de lifecycle, input, interação e gameplay durante transition, o UnityPlayerInputGateAdapter permite que um PlayerInput de gameplay obedeça esses blockers, o ObjectResetGroupTrigger permite resetar vários ResetSubject em sequência, e o ActivityRestartTrigger usa ResetSelectionConfig para resetar subjects e reiniciar a Activity atual em uma única janela visual de transition. Pause continua separado: superfície visual residente na UIGlobal, requests por UI, toggle por tecla via uma única ação Global/Pause e pausa básica da simulação com Time.timeScale = 0.

Configurando a UIGlobal

A FG_UIGlobal deve conter as superfícies globais do framework. No Preview 12, o fluxo validado usa Loading, Transition, Transition Gate, Pause visual residente e Pause por Global/Pause. A Pause Surface não é opcional neste fluxo: ela confirma o estado pausado junto com Time.timeScale = 0.

Transition Gate no Preview 12

Transition Gate é a política de bloqueio durante janelas de Route/Activity transition. Ele não é Pause, não usa Time.timeScale, não substitui Global/Pause e não torna PauseKeepUiActionMap uma decisão canônica.

AssetConfiguração recomendada para First Game
FG_GameplayRouteTransition Gate = InputInteractionAndGameplay
FG_Activity_AVisual Transition Mode = Fade
Transition Gate = InputInteractionAndGameplay
FG_Activity_BVisual Transition Mode = Fade
Transition Gate = InputInteractionAndGameplay
TransitionFadeSurfaceBlock Raycasts When Visible = true

Smoke esperado: clique Go Activity B e, durante o fade, clique novamente. O segundo clique não deve chegar como IgnoredAlreadyActive. Após a transition, os botões voltam a funcionar e Restore Activity A continua operacional.

Loading / Transition Semantics

Transition e Loading são conceitos separados. Transition cobre a janela visual de troca; Loading cobre carga real de cena. Um fluxo pode ter transition sem loading.

FluxoTransitionLoadingLeitura correta
Route switch com scene loadSucceededWithUnitySurfaceSucceededWithUnitySurfaceTroca de Route com cena carregada e superfícies Unity aplicadas.
Activity switch sem scene loadSucceededWithUnitySurfaceSkippedNoSceneLoadCorreto. Não é erro; a Activity mudou sem carregar cena.
Activity RestartPode usar transitionAparece só se houver scene load ou release side-effectRestart usa reset + clear + reenter; loading não é obrigatório.
ℹ️
Pause durante transition/loading

Durante transition ou loading, Pause é rejeitado pela policy com status='Rejected' e policyStatus='RejectedTransitionInProgress'. O request não abre PauseSurface, não muda Time.timeScale e não fica enfileirado para depois.

ℹ️
Logs após D2

Info é resumo operacional. Debug é diagnóstico técnico. Trace é waiting, retry, polling e lifecycle ruidoso de adapter.

Modelo de Pause no Preview 12

Pause é um estado lógico global da aplicação com um efeito concreto de simulação. Ele não é uma Route, não é uma Activity e não é um menu de opções completo. O runtime de Pause produz um snapshot que atualiza a superfície visual, os Gate blockers e o Time.timeScale.

⚠️
Não modele Pause como Activity

Activity continua sendo estado local dentro de uma Route. Pause fica acima disso: ele pode pausar enquanto qualquer Route/Activity está ativa, sem trocar cena e sem limpar Activity.

Pause por UI Canvas

Use PauseRequestTrigger quando a origem do Pause for um botão, menu, QA Canvas ou outro Unity Event.

FG_UIGlobal
└── PauseControls
    ├── PauseRequestTrigger
    └── Button_Pause

PauseSurface/PausePanel
└── Button_Resume
  1. Crie um GameObject PauseControlsEle pode ficar na FG_UIGlobal, fora do painel visual.
  2. Adicione PauseRequestTriggerPreencha Reason com algo legível, por exemplo firstgame.pause.ui.
  3. Configure o botão PauseNo Button_Pause → OnClick, chame PauseRequestTrigger.TogglePause().
  4. Configure o botão ResumeNo botão dentro do PausePanel, chame PauseRequestTrigger.RequestResume().
✅
Botão Pause deve usar Toggle

Para um botão genérico de Pause, prefira TogglePause(). Se usar RequestPause() enquanto já está pausado, o runtime responde corretamente com IgnoredNoChange.

Pause por tecla com Unity Input System

O Preview 12 usa PauseInputActionTrigger como caminho simples para Pause por teclado ou controle. Ele escuta uma única ação global do Unity Input System, envia um request direto ao PauseRuntime e deixa troca de mapa do PlayerInput como efeito opcional, não como requisito do Pause.

FG_UIGlobal
└── PauseInput
    └── FG_PauseKeyboard
        └── PauseInputActionTrigger

Input Actions mínimo

Crie ou atualize o asset de Input Actions, por exemplo InputSystem_Actions.inputactions, com uma ação global de Pause:

Action MapActionBindingMotivo
GlobalPause<Keyboard>/escapePausar ou resumir por um comando sistêmico único.
⚠️
Não duplique Pause em Player e UI

O trilho canônico do Preview 12 é Global/Pause. Evite Player/Pause + UI/Pause como caminho normal; isso cria dedupe e pode interferir na UI depois do Resume.

Configurando a Menu Route

A Menu Route é a primeira tela do fluxo. Ela não precisa de Activity inicial.

  1. Configure o asset FG_MenuRoute Primary Scene = FG_Menu e Startup Activity = None.
  2. Crie um botão Start Game Na cena FG_Menu, crie um botão UGUI chamado Button_StartGame.
  3. Adicione RouteRequestTrigger No botão ou em um GameObject auxiliar, adicione RouteRequestTrigger com Target Route = FG_GameplayRoute e Reason = firstgame.start.game.
  4. Conecte o OnClick No Button → On Click(), arraste o GameObject com RouteRequestTrigger e selecione RouteRequestTrigger.RequestRoute().

FIRSTGAME-3 — PlayerPrototype e Object Reset

O First Game usa um PlayerPrototype temporário no consumidor para provar gameplay mínimo, Pause, Transition, Object Reset, Reset Group, Runtime Objects e Activity Restart. Esse mover não é o sistema oficial de Player/Actor/Movement do framework.

UnityResetSubjectAdapter

O adapter é a faceta de reset do objeto. Ele registra um ResetSubject no ResetRegistry e descobre os participantes locais.

Subject Id = firstgame.player
Id Generation = AuthoredStableId
Scope = Activity
Display Name = First Game Player
Participant Discovery = Children
Include Inactive Participants = true
Include Unity Resettable Components = true

Use Scope = Activity quando o objeto pertence à Activity atual. Use Scope = Route somente quando o objeto deve sobreviver/participar do escopo da Route inteira. Objetos runtime instanciados, como caixas de teste, usam Id Generation = RuntimeInstanceId e um prefixo como firstgame.runtime.box.

Built-in participants

Use participants prontos quando o reset é simples e visualmente authorável.

UnityTransformResetParticipant
  Reset Position = true
  Reset Rotation = true
  Reset Scale = true

UnityGameObjectActiveResetParticipant
  Reset Active State = true

O UnityTransformResetParticipant é ideal para player temporário, caixas, props e objetos de puzzle que precisam voltar para uma posição/rotação/escala conhecida.

Gameplay component mode: IUnityResettable

Para objetos grandes, como player, NPC ou puzzle controller, não crie um participant separado para cada estado. O script de gameplay implementa IUnityResettable e o UnityResetSubjectAdapter registra esse componente como participant internamente.

using Immersive.Framework.Reset;
using Immersive.Framework.Reset.Unity;
using UnityEngine;

public sealed class FirstGamePlayerResettableState : MonoBehaviour, IUnityResettable, IUnityResettableMetadata
{
    private int _resetCount;
    private int _state;

    public string ResetParticipantId => "firstgame.player.resettable-state";
    public ResetParticipantRequiredness ResetRequiredness => ResetParticipantRequiredness.Required;
    public int ResetOrder => 100;
    public string ResetDisplayName => "FirstGame Player Resettable State";
    public string ResetSource => nameof(FirstGamePlayerResettableState);
    public string ResetReason => "firstgame.player.resettable-state";

    public ResetParticipantResult Reset(ResetContext context)
    {
        _resetCount++;
        _state = 0;

        Debug.Log($"[FIRSTGAME_RESETTABLE] id='{ResetParticipantId}' resetCount='{_resetCount}' state='{_state}' reason='{context.Reason}'");

        return ResetParticipantResult.CreateSucceeded(
            context.Participant,
            nameof(FirstGamePlayerResettableState),
            context.Reason,
            $"FirstGame player resettable state executed. resetCount='{_resetCount}'.");
    }
}

Critério de smoke atual: PlayerPrototype sozinho registra subjects='1' participants='2'. Com uma FG_RuntimeBox spawnada, o Reset Room deve registrar subjects='2' participants='3'. Com duas boxes, subjects='3' participants='4'.

O reset não usa ObjectEntryDeclaration nem snapshot de ObjectEntry como target. O alvo canônico é o ResetSubject.

Reset: papéis e superfície de authoring

Depois do Preview 12, o reset tem um processo comum e superfícies diferentes de authoring. O processo comum é: selecionar ResetSubject, criar ResetExecutionRequest, executar participants registrados no ResetRegistry via ResetExecutor e agregar resultado. O que muda entre botões, scripts e prefabs é apenas como o subject é registrado e como a seleção é disparada.

PeçaResponsabilidadeUso no FIRSTGAME
UnityResetSubjectAdapterRegistra o objeto como ResetSubject.Fica no root do player, prefab runtime ou objeto resetável.
UnityResetParticipantBehaviourParticipant explícito authorável no Inspector.UnityTransformResetParticipant para posição/rotação/escala.
IUnityResettableComponente de gameplay participa diretamente do reset.Health, inventory runtime state, puzzle state, player flags.
ObjectResetTriggerResetar um único ResetSubject.Button_ResetPlayer.
ObjectResetGroupTriggerResetar uma seleção de subjects sem reiniciar Activity.Button_ResetRoom.
ActivityRestartTriggerResetar subjects e reiniciar a Activity atual em uma janela visual.Button_RestartActivity.
⚠️
Não empilhe triggers no mesmo OnClick

Um botão deve chamar apenas a superfície desejada. Button_ResetPlayer chama só ObjectResetTrigger, Button_ResetRoom chama só ObjectResetGroupTrigger, e Button_RestartActivity chama só ActivityRestartTrigger. Se o mesmo botão chamar Reset Group e Restart, o reset parecerá duplicado.

FIRSTGAME-4 — Object Reset Group

O ObjectResetGroupTrigger é uma ferramenta para resetar vários ResetSubject em sequência. Ele executa uma ResetSelectionConfig pelo ResetExecutor. No FIRSTGAME, ele deve pegar player + runtime boxes + componentes IUnityResettable descobertos pelo adapter.

ℹ️
Reset Group não é Restart

O Reset Group não reinicia Activity, não recarrega cena, não refaz spawn e não toca em Save. Ele apenas executa uma seleção de ResetSubjects.

Canvas_Gameplay
└── Button_ResetRoom
    └── ObjectResetGroupTrigger
        ├── Group Id = firstgame.reset.room
        └── Selection
            ├── Mode = CurrentRouteAndActivitySubjects
            ├── Allow No Subjects = false
            ├── Allow No Participants = false
            ├── Stop On Failure = true
            └── Yield Between Subjects = false

ObjectResetGroupTrigger

Use este componente quando um botão ou evento precisa resetar um conjunto de objetos.

Group Id = firstgame.reset.room
Reason = firstgame.reset.room
Selection
  Mode = CurrentRouteAndActivitySubjects
  Allow No Subjects = false
  Allow No Participants = false
  Stop On Failure = true
  Yield Between Subjects = false

Para grupos locais de cena, prefira referência direta ao UnityResetSubjectAdapter. Para runtime ou referências indiretas, use texto de Reset Subject Id.

FIRSTGAME-5 — Activity Restart via Reset Selection Policy

O ActivityRestartTrigger cria um restart simples de Activity sem depender de um ObjectResetGroupTrigger. Ele usa o executor comum de reset, escolhe alvos por policy e depois executa Clear/Re-enter como um fluxo composto com uma única transição visual.

⚠️
Activity Restart não é CycleReset completo

Este corte não cria lifecycle de Player/Actor, não recarrega Route, não descobre participants automaticamente e não substitui o sistema futuro de Restart/CycleReset. Ele é uma composição útil para sala, puzzle ou encounter.

Canvas_Gameplay
└── Button_RestartActivity
    └── ActivityRestartTrigger
        ├── Target Activity = vazio para usar Activity atual
        ├── Use Current Activity When Target Missing = true
        ├── Require Target Activity Is Current = true
        ├── Reason = firstgame.restart.activity
        └── Reset Selection
            ├── Mode = CurrentRouteAndActivitySubjects
            ├── Allow No Subjects = false
            ├── Allow No Participants = false
            ├── Stop On Failure = true
            └── Yield Between Subjects = false

Reset Selection Mode

A policy responde uma pergunta: quais ResetSubjects entram neste reset? O processo de reset é o mesmo para todos os modos.

ModoUso recomendado
ExplicitSubjectsLista manual no trigger. Use quando a sala é fixa e você quer evitar seleção dinâmica.
CurrentActivitySubjectsSubjects próprios da Activity atual, como o firstgame.player quando o adapter está em Scope = Activity.
CurrentRouteSubjectsSubjects Route-scoped atuais, caso o jogo tenha objetos que sobrevivem à Activity.
CurrentRouteAndActivitySubjectsModo usado no FIRSTGAME atual para pegar Player + Runtime Boxes no mesmo Reset Room/Restart.
AllCurrentSubjectsTodos os subjects conhecidos no ResetRegistry atual. Use com cuidado.

Ordem de execução

ActivityRestartTrigger.RequestActivityRestart()
  1. Resolve ResetSelectionConfig
  2. FrameworkRuntimeHost.RestartActivityAsync(activity, beforeRestartLifecycle)
     2.1 transitionBefore fecha o fade
     2.2 ResetExecutor executa os subjects selecionados
     2.3 limpa a Activity atual internamente
     2.4 reentra na mesma Activity internamente
     2.5 transitionAfter abre o fade

O reset roda dentro da janela visual do restart, depois do fade fechar e antes do Clear/Re-enter. O Clear/Re-enter não deve aparecer como dois fades separados; os status clearStatus e reenterStatus ficam agregados no resultado final do restart. Loading só deve aparecer se houver scene load ou release side-effect; restart sem scene load não precisa de Loading.

ℹ️
Como saber que o fade duplo foi corrigido

Com Debug habilitado, o restart mostra Activity Restart reset stage completed inside restart transition antes de Activity Restart Request completed. Em Info, deve aparecer apenas a conclusão pública do restart. Não deve aparecer como duas requests públicas separadas com reason='firstgame.restart.activity:clear' e reason='firstgame.restart.activity:reenter'. Esses detalhes agora são internos ao fluxo composto.

Fase avançada — Reset programático

Esta fase é para programadores. O authoring canônico continua sendo pelo Inspector, mas scripts do jogo podem criar objetos resetáveis, chamar triggers e reagir aos resultados sem acessar diretamente o ResetRegistry.

⚠️
Limite público atual

Em código de jogo, prefira chamar ObjectResetTrigger, ObjectResetGroupTrigger e ActivityRestartTrigger. Não acesse ResetRegistry diretamente no projeto consumidor. O registry é infraestrutura do framework; os triggers são a API pública de request.

1. Criar um runtime object resetável por código

O prefab deve conter UnityResetSubjectAdapter com Id Generation = RuntimeInstanceId, prefixo estável e pelo menos um participant ou componente IUnityResettable. O script apenas instancia o prefab; o adapter registra o subject no OnEnable.

using UnityEngine;

public sealed class FirstGameRuntimeObjectSpawner : MonoBehaviour
{
    [SerializeField] private GameObject runtimeBoxPrefab;
    [SerializeField] private Transform spawnPoint;
    [SerializeField] private string spawnReason = "firstgame.runtime.box";

    private int _spawnCount;

    public GameObject Spawn()
    {
        if (runtimeBoxPrefab == null)
        {
            Debug.LogWarning("Runtime box prefab is missing.");
            return null;
        }

        Vector3 position = spawnPoint != null ? spawnPoint.position : transform.position;
        Quaternion rotation = spawnPoint != null ? spawnPoint.rotation : Quaternion.identity;
        GameObject instance = Instantiate(runtimeBoxPrefab, position, rotation);
        _spawnCount++;
        instance.name = $"{runtimeBoxPrefab.name} (Runtime {_spawnCount})";

        Debug.Log($"[FIRSTGAME_RUNTIME_OBJECT] spawned. reason='{spawnReason}' prefab='{runtimeBoxPrefab.name}' instance='{instance.name}' count='{_spawnCount}'");
        return instance;
    }
}

2. Disparar reset por script usando triggers

Configure os triggers no Inspector e use um script fino para chamar os métodos públicos. Isso permite conectar input, UI customizada, dev hotkeys ou ferramentas internas sem duplicar a lógica de reset.

using Immersive.Framework.ActivityRestart;
using Immersive.Framework.ObjectReset;
using Immersive.Framework.Reset;
using UnityEngine;

public sealed class FirstGameResetActions : MonoBehaviour
{
    [SerializeField] private ObjectResetTrigger resetPlayer;
    [SerializeField] private ObjectResetGroupTrigger resetRoom;
    [SerializeField] private ActivityRestartTrigger restartActivity;

    public void ResetPlayer()
    {
        resetPlayer?.RequestObjectReset();
    }

    public async void ResetRoom()
    {
        if (resetRoom == null)
        {
            return;
        }

        ResetExecutionResult result = await resetRoom.RequestObjectResetGroupAsync();
        Debug.Log($"[FIRSTGAME_RESET] room status='{result.Status}' subjects='{result.SubjectCount}' participants='{result.ParticipantCount}'");
    }

    public async void RestartActivity()
    {
        if (restartActivity == null)
        {
            return;
        }

        ActivityRestartResult result = await restartActivity.RequestActivityRestartAsync();
        Debug.Log($"[FIRSTGAME_RESET] restart status='{result.Status}' resetParticipants='{result.ResetParticipantCount}' clear='{result.ClearStatus}' reenter='{result.ReenterStatus}'");
    }
}

Use async void apenas em fronteiras Unity/OnClick/ContextMenu. Em sistemas próprios, prefira encapsular o fluxo em métodos que retornem Awaitable quando o contexto permitir.

3. Reagir a resultado sem ler console

Os triggers mantêm o último resultado. Isso é útil para UI, debug overlays e validação manual.

using Immersive.Framework.ObjectReset;
using UnityEngine;

public sealed class FirstGameResetStatusLabel : MonoBehaviour
{
    [SerializeField] private ObjectResetGroupTrigger resetRoom;

    public void PrintLastResetRoomResult()
    {
        if (resetRoom == null || !resetRoom.HasLastResult)
        {
            Debug.Log("Reset Room ainda não rodou.");
            return;
        }

        Debug.Log($"Reset Room last status='{resetRoom.LastExecutionStatus}' participants='{resetRoom.LastParticipantCount}' succeeded='{resetRoom.LastSucceededParticipantCount}' failed='{resetRoom.LastFailedParticipantCount}'");
    }
}

FIRSTGAME-3B — Unity PlayerInput Gate Adapter

Gate lógico versus efeito Unity

O Gate levanta a regra de bloqueio. O adapter aplica essa regra no Unity desabilitando o action map Player. Assim o FirstGamePlayerMover continua simples, mas deixa de receber Player/Move durante Pause ou Transition Gate.

Erros comuns e correções

🔴 Dois AudioListeners ou dois EventSystems

Causa comum: FG_UIGlobal e a Route Scene têm câmera/EventSystem ao mesmo tempo, ou o package é anterior ao Preview 6 e duplica a UIGlobal aberta no Editor.
Solução: use v1.0.0-preview.12; mantenha apenas um AudioListener e um EventSystem no runtime.

🔴 "Unloading the last loaded scene is not supported"

Causa: versão antiga do validator tentando fechar a última cena aberta ao validar a UIGlobal.
Solução: atualizar para v1.0.0-preview.12. O validator agora trata ownership de cena com segurança.

⚠️ Loading parece não animar

Causa provável: cena leve demais; o progresso vai de 0 a 1 quase instantaneamente.
Solução: se o log mostra loadingProgressPercent='100' e sem failures, está funcional.

⚠️ Escape não pausa

Causa comum: Input Actions sem Global/Pause, PlayerInput ou Actions Asset não atribuído, ou uso do bridge avançado sem os targets exigidos.
Solução: para FIRSTGAME, use PauseInputActionTrigger, configure Pause Action Map Name = Global e Pause Action Name = Pause. Não use Player/Pause + UI/Pause como trilho canônico.

⚠️ FailedBridgeRequest / FailedPreflight no Pause input

Causa: o caminho PauseInputActionRuntimeBridgeTrigger + PauseInputModeUnityPlayerInputRuntimeBridge exige preflight de InputMode/PlayerInput e não é o caminho simples do FIRSTGAME.
Solução: substitua pelo PauseInputActionTrigger quando o objetivo for apenas Escape → Toggle Pause.

⚠️ Activity não aparece

Causa comum: FG_GameplayRoute sem Startup Activity, ou ActivityLocalVisibilityAdapter apontando para a Activity errada.
Solução: confirme Startup Activity = FG_Activity_A e os adapters com Local Content Id explícito.

⚠️ Segundo clique durante fade chega no runtime

Causa comum: Activity com Transition Gate = None/LifecycleRequestsOnly ou UnityFadeCurtainEffectAdapter sem bloquear raycasts quando visível.
Solução: para First Game, use InputInteractionAndGameplay nas Activities com Fade e mantenha Block Raycasts When Visible ativo no Transition surface.

⚠️ Movimento ainda funciona durante Transition Gate

Causa comum: o script de movimento lê InputAction diretamente e não participa do Gate.
Solução: adicione UnityPlayerInputGateAdapter no PlayerPrototype e configure Gameplay Action Map Name = Player, Block On Input Acceptance = true e Block On Gameplay Action = true.

⚠️ Restart Activity faz dois fades

Causa: versão intermediária do Preview 12 executava ActivityClear e Activity como duas requests visuais públicas.
Solução: use o patch de Activity Restart com transição única. O log não deve mostrar firstgame.restart.activity:clear e firstgame.restart.activity:reenter como duas requests públicas separadas.

⚠️ Restart parece resetar duas vezes

Causa comum: o mesmo botão chama ObjectResetGroupTrigger.RequestObjectResetGroup() e também ActivityRestartTrigger.RequestActivityRestart().
Solução: no botão de restart, deixe apenas ActivityRestartTrigger e apenas uma chamada no Button.OnClick.

O que não fazer neste Preview

🚫
Fora do escopo do FIRSTGAME Usage Model

Não edite o package Git dentro do consumidor.

Não copie assets, cenas ou ProjectSettings do projeto QA antigo.

Não abra Save, Snapshot, Player, Actor, Inventory ou gameplay complexo.

Não crie service locator, singleton ou manager genérico.

Não use SceneManager direto para trocar telas do jogo. Use RouteRequestTrigger e os assets do framework.

Não trate Pause como Activity ou Route.

Não use Input.GetKeyDown solto como caminho canônico de Pause.

Não trate UnityPlayerInputGateAdapter como Player/Actor lifecycle; ele é apenas um adapter opt-in de Gate para o Unity Input System.

Não use o bridge avançado de InputMode quando o objetivo for apenas Escape → Toggle Pause.

Time.timeScale é efeito básico do Pause no Preview 12, mas não substitui o contrato do PauseRuntime. Não escreva scripts paralelos mudando Time.timeScale fora do runtime de Pause.
